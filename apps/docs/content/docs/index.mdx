## Syntax at a Glance

```wire
(element-type "content" :properties... children...)
```

| Part | What it does | Example |
|------|--------------|---------|
| `element-type` | What kind of element | `box`, `text`, `button`, `card` |
| `"content"` | Optional text content | `"Hello World"` |
| `:property` | Boolean (flag) | `:primary`, `:center` |
| `:property value` | Key-value pair | `:gap 16`, `:to dashboard` |
| `children` | Nested elements | `(box (text "hi"))` |

## Core Concepts

### Screens & Navigation

Every document contains screens. Use `:to` to navigate between them or open overlays:

```wire preview:300
(wire
  (screen main "Main" :desktop
    (box :center :gap 12 :padding 24
      (text "Navigation Demo" :high)
      (button "Go to Settings" :primary :to settings)
      (button "Open Modal" :to #confirm))

    (modal :id "confirm"
      (box :gap 16 :padding 24
        (text "Confirm Action" :high)
        (box :row :gap 8
          (button "Cancel" :ghost :to :close)
          (button "Confirm" :primary)))))

  (screen settings "Settings" :desktop
    (box :center :gap 12 :padding 24
      (text "Settings" :high)
      (button "Back to Main" :ghost :to main))))
```

### Reusable Components

Define components to avoid repetition:

```wire preview:250
(wire
  (define stat-card (label value)
    (card :padding 16 :gap 4
      (text $label :low)
      (text $value :high)))

  (screen main "Stats" :desktop
    (box :row :gap 16 :padding 24
      (stat-card :label "Users" :value "1,234")
      (stat-card :label "Revenue" :value "$5,678")
      (stat-card :label "Orders" :value "890"))))
```

### Shared Layouts

Share common page structure across screens:

```wire preview:350
(wire
  (layout app-layout
    (box :row :full
      (nav :gap 4 :padding 16 :width "180px"
        (text "MyApp" :high)
        (divider)
        (button "Home" :ghost :full :start :to home)
        (button "Settings" :ghost :full :start :to settings))
      (slot)))

  (screen home "Home" :desktop :layout app-layout
    (box :gap 16 :padding 24
      (text "Home Page" :high)
      (text "Welcome to the app" :low)))

  (screen settings "Settings" :desktop :layout app-layout
    (box :gap 16 :padding 24
      (text "Settings" :high)
      (text "Configure your preferences" :low))))
```

## FAQ

<Accordions>
<Accordion title="How is this different from Figma or Sketch?">

Figma and Sketch are visual editors — you drag boxes and click to style. WireScript is a **text-based DSL** — you write code that describes structure.

| | Figma/Sketch | WireScript |
|---|---|---|
| **Editing** | Visual drag-and-drop | Text editor |
| **Version control** | Binary files, merge conflicts | Clean diffs, easy merges |
| **AI generation** | Plugin-dependent | Native — LLMs write it directly |
| **Documentation** | Separate from specs | Embeds in Markdown |
| **Use case** | High-fidelity design | Rapid wireframing |

WireScript isn't a replacement — it's for the spec phase before visual design begins.

</Accordion>
<Accordion title="How is this different from AI wireframe tools?">

Tools like [UX Pilot](https://uxpilot.ai/), [Visily](https://www.visily.ai/), and [Miro AI](https://miro.com/ai/wireframe/) use natural language prompts to generate visual wireframes. WireScript takes a different approach:

- **Structured DSL vs. natural language** — WireScript's S-expression syntax is unambiguous and deterministic
- **Text output vs. visual output** — WireScript produces readable code you can edit, version, and embed
- **Human + AI workflow** — Write it yourself or let AI generate it; the syntax works either way

Prompt-based tools are great for quick exploration. WireScript is better when you need wireframes that live in your codebase alongside documentation.

</Accordion>
<Accordion title="Can AI generate valid WireScript?">

Yes. The S-expression syntax is simple and unambiguous — exactly what LLMs excel at. Claude, GPT, Gemini, and other models generate valid WireScript reliably because:

- **Consistent structure** — Every element follows `(type content? :properties... children...)`
- **No ambiguity** — Keywords like `:primary`, `:col`, `:gap 16` have fixed meanings
- **Self-contained** — No external dependencies or imports to manage

Ask any LLM to "draw a login form in WireScript" and it will produce working code.

</Accordion>
<Accordion title="What interactivity is supported?">

WireScript supports navigation and overlays for building complete user flows:

```wire
; Navigate to another screen
(button "Dashboard" :to dashboard)

; Open a modal or drawer
(button "Settings" :to #settings-modal)

; Close the current overlay
(button "Cancel" :to :close)
```

Screens can share layouts with auto-active navigation — sidebar links automatically highlight when you're on that screen.

</Accordion>
<Accordion title="Can I customize the styling?">

WireScript intentionally renders with a neutral, consistent style. Wireframes should communicate **structure**, not aesthetics.

What you can control:
- Layout (`:row`, `:col`, `:grid`, `:gap`, `:padding`)
- Sizing (`:width`, `:height`, `:full`, `:fit`)
- Emphasis (`:high`, `:medium`, `:low`)
- Variants (`:primary`, `:ghost`, `:danger`)
- Viewport (`:mobile`, `:tablet`, `:desktop`)

For pixel-perfect designs, export to PNG and use as references for your design system.

</Accordion>
<Accordion title="How do I embed WireScript in Markdown?">

Use fenced code blocks with the `wire` language identifier:

````markdown
```wire
(box :gap 16
  (text "Hello" :high)
  (button "Click me" :primary))
```
````

In documentation systems that support it (like this site), add `preview` to render a live preview:

````markdown
```wire preview
(button "Live Preview" :primary)
```
````
